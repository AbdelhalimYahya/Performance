# How to Detect Frontend Performance Issues

> A senior engineer's reference for identifying, measuring, and diagnosing frontend performance problems in production and development environments.

---

## 1. The Performance Problem Space

Before touching a single line of code, you need to understand that "performance" is not one thing. There are three distinct categories, and confusing them leads to optimizing the wrong problems.

**Perceived Performance** is what the user *feels*. A skeleton screen with a loading spinner feels faster than a blank white screen for 2 seconds, even if the total data transfer is identical. Perceived performance is about feedback loops — the browser must show *something* meaningful as fast as possible.

**Actual Performance** is what the metrics say. LCP, TTI, total blocking time, bundle size, network waterfalls. These are objective, measurable, and reproducible. They matter for SEO, for accessibility, and for users on slow connections.

**User-Impacting Performance** is the intersection: what actually degrades the experience for real people. A 200ms delay on a checkout button matters enormously. A 200ms delay on a footer animation nobody scrolls to matters not at all.

The trap most engineers fall into is optimizing actual performance metrics without considering whether users notice or care. Profile first, measure second, fix third — always in that order.

## 2. Core Web Vitals Deep Dive

Core Web Vitals are the three metrics Google uses to evaluate page experience. They are not optional for production sites — they directly affect search ranking.

### Largest Contentful Paint (LCP)

**What it measures:** Time from navigation start to when the largest visible element (image, video poster, or block-level text) finishes rendering.

**Thresholds:**
- Good: ≤ 2,500ms
- Needs Improvement: 2,500ms – 4,000ms
- Poor: > 4,000ms

**Common causes of failure:**
- Slow server response (high TTFB)
- Render-blocking resources (CSS, synchronous JS)
- Large unoptimized images without `loading="lazy"` or `fetchpriority`
- Client-side rendering where the LCP element is generated by JavaScript
- Web fonts delaying text rendering (FOIT)

**How to read it:** Use Chrome DevTools → Performance tab → take a recording → look for the "Largest Contentful Paint" marker. It will show you the exact element. Click it to see what the browser considered "largest."

**Measurement tip:** LCP is measured from the *largest* element, not the *most important* one. A hero image that loads fast but is smaller in pixel area than a below-fold banner will not be the LCP element.

### Interaction to Next Paint (INP)

**What it measures:** Latency of the slowest interaction (click, tap, keypress) during the page lifecycle. It replaces FID as the responsiveness metric.

**Thresholds:**
- Good: ≤ 200ms
- Needs Improvement: 200ms – 500ms
- Poor: > 500ms

**Common causes of failure:**
- Long JavaScript tasks blocking the main thread
- Expensive event handlers (scroll, click) doing synchronous work
- Layout thrashing from forced synchronous layouts
- Third-party scripts (analytics, chat widgets) executing on the main thread

**How to read it:** INP is an *overall* responsiveness score, not a single measurement. Use `web-vitals` library's `onINP()` callback to capture every interaction. The worst one determines your INP.

### Cumulative Layout Shift (CLS)

**What it measures:** Sum of all unexpected layout shifts that occur during the page lifecycle. Each shift is scored by its impact (how many pixels moved) × distance (how far elements shifted).

**Thresholds:**
- Good: ≤ 0.1
- Needs Improvement: 0.1 – 0.25
- Poor: > 0.25

**Common causes of failure:**
- Images without `width` and `height` attributes
- Ads or embeds injecting content dynamically
- Web fonts swapping (FOUT) causing text reflow
- Content injected above the fold after initial render
- Dynamic banners or cookie consent overlays

**How to read it:** In DevTools → Performance → take a recording → look for "Layout Shift" events in the experience row. Click each one to see which elements moved and by how many pixels.

### Time to First Byte (TTFB)

**What it measures:** Time from navigation start to the first byte of the HTML document received from the server.

**Thresholds:**
- Good: ≤ 800ms
- Needs Improvement: 800ms – 1,800ms
- Poor: > 1,800ms

**Common causes of failure:**
- Slow server processing (database queries, template rendering)
- No CDN or CDN configured incorrectly
- No HTTP/2 or HTTP/3
- DNS resolution delays
- TLS handshake overhead

### First Contentful Paint (FCP)

**What it measures:** Time from navigation start to the first DOM content being rendered (text, image, canvas).

**Thresholds:**
- Good: ≤ 1,800ms
- Needs Improvement: 1,800ms – 3,000ms
- Poor: > 3,000ms

**Common causes of failure:**
- Render-blocking CSS or JavaScript
- Slow TTFB (FCP cannot be faster than TTFB)
- Large critical CSS not inlined

## 3. Browser DevTools Profiling

### Performance Tab: Flame Chart Reading

The Performance tab records everything the browser does for a few seconds. The flame chart is the most important visualization:

```
Main Thread
├─ Scripting    ████████░░░░░░░░░░░░░░░░  (yellow)
├─ Rendering    ░░░░░░░░████░░░░░░░░░░░░  (purple)
├─ Painting     ░░░░░░░░░░░░████░░░░░░░░  (green)
├─ System       ░░░░░░░░░░░░░░░░░░░░░░░░  (gray)
└─ Idle         ░░░░░░░░░░░░░░░░░░░░████  (white)
```

**Reading a bad flame chart:**
```
Main Thread
├─ Scripting    ████████████████████████  ← One giant yellow block
│  └─ parseHTML ████████████████████████  ← Parser blocked by sync script
│     └─ eval   ████████████████████████  ← Third-party analytics
├─ Rendering    (nothing — blocked)
├─ Painting     (nothing — blocked)
└─ Idle         (nothing — blocked)

Result: Main thread blocked for 4.2 seconds
```

The key insight: if the main thread is yellow (scripting) for long stretches, JavaScript is your problem. If it's purple (rendering), your DOM is too complex or you're triggering expensive layouts. If it's white (idle) with long gaps between tasks, you have nothing to worry about.

### Long Tasks

Any task exceeding 50ms on the main thread is a "long task." Chrome DevTools marks these with a red triangle in the Performance tab. The `PerformanceObserver` API can detect them programmatically:

```javascript
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.duration > 50) {
      console.warn(`Long task: ${entry.duration.toFixed(1)}ms`, entry);
    }
  }
});
observer.observe({ type: 'longtask', buffered: true });
```

### Layout Thrashing

Layout thrashing happens when JavaScript reads a layout property (like `offsetHeight`) and then immediately writes to the DOM, forcing the browser to recalculate layout before the next read. In a flame chart, this appears as rapid alternation between yellow (scripting) and purple (rendering) blocks.

Detect it by looking for forced synchronous layout warnings in the Performance panel, or by using the Layout Instability API.

## 4. Lighthouse Audit

### How to Run It

1. **Chrome DevTools:** F12 → Lighthouse tab → Generate report
2. **CLI:** `npx lighthouse https://yoursite.com --output=html --view`
3. **CI/CD:** `npx lhci autorun` (using `@lhci/cli`)

### Reading the Report

**Opportunities** are actionable suggestions with estimated savings. "Eliminate render-blocking resources" tells you *what* to fix and *how much* time you'd save. Prioritize these.

**Diagnostics** are informational. "Reduce unused JavaScript" doesn't give a time estimate — it tells you there's potential waste. Useful but lower priority.

**Passed audits** means nothing to fix. Don't celebrate too early — Lighthouse only tests what it can automate.

### When Lighthouse Lies

- **Network throttling is simulated.** Real 3G is worse than Lighthouse's "Slow 3G" preset.
- **It runs on a clean profile.** No extensions, no ad blockers, no cached data from previous visits.
- **Single page, single run.** Results vary between runs. Always run 3+ times and average.
- **It can't measure user experience.** Lighthouse doesn't know about your cookie consent modal or your interstitial ad.
- **JavaScript-heavy SPAs** may score well on Lighthouse but feel terrible on low-end devices because Lighthouse runs on a powerful desktop with fast CPU.

Use Lighthouse for **relative comparison** (before vs. after optimization), not absolute truth.

## 5. React DevTools Profiler

### Setup

1. Install React DevTools browser extension
2. Open DevTools → Profiler tab
3. Click the record button (circle) → interact with your app → stop recording

### Ranked Chart

The "Ranked" chart shows each component that rendered, sorted by time (slowest first). Each bar represents a single render. Width = time spent. This is where you find your bottlenecks.

### Reading Render Times

- **Mount phase:** Component first appears. Expected to be slow.
- **Update phase:** Component re-rendered. Should be fast. If it's slow, investigate why.
- **Wasted renders:** Component rendered but produced the same output. Shown in the Ranked chart as components that rendered but didn't change visually.

### What Wasted Renders Look Like

A parent component updates state that its children don't use. Every child re-renders even though their props haven't changed. In the profiler, you'll see multiple components with identical output before and after — that's waste.

Use `React.memo()`, `useMemo()`, and `useCallback()` to prevent these. But measure first — premature memoization can make things worse.

## 6. Network Waterfall Analysis

### Reading the Waterfall

The Network tab in DevTools shows every request as a horizontal bar. The bar has segments:

```
DNS Lookup   ██████
TCP          ░░░░░░██████
TLS          ░░░░░░░░░░░░██████
Request      ░░░░░░░░░░░░░░░░░░██
Waiting      ░░░░░░░░░░░░░░░░░░░░░░████████
Content      ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░████
```

**Waiting (TTFB):** Time from request sent to first byte received. This is server processing time.

**Content Download:** Time to transfer the full response. Depends on size and connection speed.

### Render-Blocking Resources

Resources that block rendering appear as sequential bars before any content paint. Look for:
- CSS files loaded in `<head>` without `media` attribute
- Synchronous `<script>` tags without `defer` or `async`
- Fonts loaded without `font-display: swap`

### Critical Path

The critical path is the sequence of requests that must complete before the page becomes visible. To find it:

1. Look for the FCP marker in the Performance timeline
2. Trace backward: what resources were being downloaded when FCP fired?
3. Those are your critical resources. Optimize them first.

## 7. Real User Monitoring vs Synthetic

### Real User Monitoring (RUM)

Collects metrics from actual users in real conditions. You see real device performance, real network conditions, real usage patterns.

**Pros:**真实数据, covers all devices/networks, finds real problems
**Cons:** Harder to set up, noisy data, requires statistical significance, can't control variables

**Tools:** Google Analytics (web-vitals), SpeedCurve, Sentry Performance, Datadog RUM, LogRocket

### Synthetic Monitoring

Tests your site from controlled environments with fixed conditions. Consistent, reproducible, fast feedback.

**Pros:** Reproducible, fast feedback loop, easy to set up baselines
**Cons:** Doesn't reflect real users, limited device/network coverage, misses interaction patterns

**Tools:** Lighthouse CI, WebPageTest, Pingdom, GTmetrix

### Comparison Table

| Aspect | RUM | Synthetic |
|--------|-----|-----------|
| Data source | Real users | Automated scripts |
| Environment | Uncontrolled | Controlled |
| Cost | Higher (data storage) | Lower |
| Setup complexity | High | Low |
| Statistical noise | High | Low |
| Use case | Production monitoring | Pre-deploy validation |
| Device coverage | All devices | Limited set |
| Interaction testing | Full user flows | Scripted scenarios |

**Recommendation:** Use both. Synthetic for CI/CD gates and pre-deploy checks. RUM for production monitoring and discovering issues you didn't anticipate.

## 8. Performance Budget

A performance budget is a set of limits on metrics that your build process enforces. If a change makes things worse, the build fails.

### Setting a Budget

Decide on thresholds based on your users and business requirements:

```json
{
  "budgets": [
    {
      "path": "/",
      "timings": [
        { "metric": "total-byte-weight", "budget": 500000 },
        { "metric": "total-blocking-time", "budget": 300 },
        { "metric": "largest-contentful-paint", "budget": 2500 }
      ]
    }
  ]
}
```

### Enforcing in CI

Use Lighthouse CI to gate your deployments:

```yaml
# lighthouserc.js
module.exports = {
  ci: {
    assert: {
      assertions: {
        'performance': ['error', { minScore: 0.9 }],
        'largest-contentful-paint': ['error', { maxNumericValue: 2500 }],
        'cumulative-layout-shift': ['error', { maxNumericValue: 0.1 }],
        'total-blocking-time': ['error', { maxNumericValue: 300 }]
      }
    }
  }
};
```

Add this to your CI pipeline:

```bash
npx lhci autorun
```

If any assertion fails, the CI build fails. This prevents performance regressions from reaching production.

## 9. Quick Checklist

Before calling a page "fast," verify all 20 items:

| # | Check | Tool | Target |
|---|-------|------|--------|
| 1 | LCP | Lighthouse / web-vitals | ≤ 2,500ms |
| 2 | INP | web-vitals / Chrome UX Report | ≤ 200ms |
| 3 | CLS | Lighthouse / DevTools | ≤ 0.1 |
| 4 | TTFB | DevTools Network | ≤ 800ms |
| 5 | FCP | Lighthouse | ≤ 1,800ms |
| 6 | Total bundle size | `npx source-map-explorer` | < 200KB gzipped |
| 7 | Unused JavaScript | Lighthouse Coverage tab | < 10% waste |
| 8 | Unused CSS | Lighthouse Coverage tab | < 10% waste |
| 9 | Image formats | Manual check | WebP/AVIF, no uncompressed PNG/JPEG |
| 10 | Image dimensions | `<img>` tags | Always have `width` and `height` |
| 11 | Font loading | Network tab | `font-display: swap` or `optional` |
| 12 | Render-blocking resources | Lighthouse | 0 render-blocking scripts |
| 13 | Third-party impact | Performance tab | Third-party JS < 100ms main thread |
| 14 | Long tasks | PerformanceObserver | 0 tasks > 50ms |
| 15 | Layout shifts | DevTools Performance | No unexpected shifts |
| 16 | HTTP/2 or HTTP/3 | Network tab protocol column | Enabled |
| 17 | Compression | Response headers | Brotli or gzip enabled |
| 18 | Caching | Response headers | Cache-Control set for static assets |
| 19 | Critical CSS | Coverage tab | Above-fold CSS inlined or preloaded |
| 20 | Mobile performance | Lighthouse mobile mode | Score ≥ 90 |
